# 리스크 엔진(Risk Engine) 입력 데이터 구조화 설계 초안

## 🎯 목표
복잡하고 비표준적인 법규 및 운영 데이터를 정량적 점수(Score)로 환산할 수 있는 최소한의 DB 스키마 정의. 이 스키마는 'Silver Tier' 상품 판매를 위한 리스크 진단 백서의 근거가 됩니다.

## 💡 핵심 원칙
1.  **원천 데이터 추적 가능성:** 모든 점수는 반드시 특정 규정/문서(Source)에 기반해야 합니다.
2.  **점수 가중치 명시:** 위험도에 따라 가중치를 부여할 수 있는 필드가 필수입니다. (Risk Weighting)

## 📊 스키마 정의: Compliance Risk Metric Table (`CR_METRIC`)
| 컬럼명 | 데이터 타입 | 설명 | 예시 값 | 비고 |
| :--- | :--- | :--- | :--- | :--- |
| `Metric_ID` | INT (PK) | 고유 식별자 (자동 생성) | RCM-001 | |
| `Regulation_Code` | VARCHAR(20) | 해당 규정/법규의 코드명 또는 출처 | GDPR Article 5 / 국내 개인정보보호법 제18조 | 현빈 에이전트가 정의 필요 (출처 명확화) |
| `Risk_Category` | VARCHAR(50) | 위험 분류 (예: 데이터 보안, 컴플라이언스, 운영 효율성) | Data Security | 필터링 용이성 확보 |
| `Violation_Description` | TEXT | 위반 항목에 대한 구체적인 설명. | "사용자 동의 없이 제3자에게 데이터를 제공함." | 백서 작성 시 활용할 문구 |
| `Impact_Severity` | ENUM(Low, Medium, High) | 해당 위반이 기업에 미치는 영향도 (심각성) | High | 영업 컨셉 자극 용도 |
| `Base_Score` | FLOAT | 기본 위험 점수. (0~10점 스케일 가정) | 8.5 | 초기 가중치 산정 기준 |
| `Weighting_Factor` | FLOAT | 사장님 판단에 따른 중요도/가중치 계수 (Multiplier). | 1.5 | 이 값이 최종 리스크 점수에 반영됨 |
| `Mitigation_Action` | TEXT | 해당 위험을 예방하기 위한 구체적인 조치사항. | '동의서 개정 및 재고지 필수' | 솔루션의 해결책 제시 근거 |

## 🔍 다음 단계 (현빈 에이전트 연계)
*   **필수 확인 사항:** 위 스키마가 실제 어떤 비즈니스 프로세스(예: 사용자 가입, 데이터 전송 등)와 연결되어 작동하는지 **최소한의 시나리오 흐름도**를 현빈 에이전트에게 요청하여 매핑해야 합니다.
*   **다음 목표:** 이 구조화된 스키마에 맞춰 실제 'Top 5' 리스크 사례의 데이터를 채워 넣는 작업을 진행할 것입니다.
