#### 3. 구조적 무결성 체크에의 활용 (Systemic Integrity Use)
이 정보가 가장 중요한 부분입니다. 우리는 단순히 '작동하는' 시스템을 만드는 게 목표가 아닙니다. 우리의 상품은 **'시스템 실패 경험(Failure Experience)'** 그 자체예요.

1.  **종속성 분석:** LLM과 이미지 모델의 연결 상태, Rate Limit, 그리고 각 API의 성공/실패 응답 코드를 모두 수집하여 '통합 모니터링 대시보드'를 구축해야 합니다. (`[근거: Self-RAG]`)
2.  **Fail State 설계:** 만약 Gemini 호출이 실패하거나(API Down), Rate Limit에 걸리면, 이 에러 상황을 고객에게 **"당신의 데이터가 처리되지 못하는 시스템적 결함"**으로 보이게 연출해야 합니다. 이것이 바로 우리가 판매할 '위협'입니다.
3.  **데이터 흐름 검증:** 모든 API 호출은 `사용자 입력 $\rightarrow$ (API 1: 리스크 데이터 수집) $\rightarrow$ (Logic Layer: 위험 등급 결정) $\rightarrow$ (API 2: Red Zone 시각화 출력)`의 순서로, 한 단계도 빠지거나 실패할 경우 즉시 경고를 띄우도록 아키텍처를 설계했습니다. (`[근거: Self-RAG]`)

**요약 및 액션 플랜:**

현재 Gemini API는 핵심 LLM 엔진으로 작동 중이나, 이미지 생성 모델의 구체적인 연동과 라이브 상태 체크가 불가능합니다. 가장 시급한 것은 이 **외부 의존성(External Dependency)** 자체를 관리하는 코드를 시스템에 삽입하는 것입니다.

지금은 단순히 "연결되어 있는지" 확인하는 걸 넘어, **'만약 끊어졌을 때 어떤 에러 코드가 뜨고 어떻게 보여줄지'**까지 설계해야 합니다.
