우선 핵심 로직을 담을 서비스를 분리하겠습니다. 이 서비스는 외부 API 호출을 시뮬레이션하고, 그 결과에 따라 게이트키퍼의 '경고 레벨'과 다음 상태를 결정합니다.

/**
 * @fileOverview yobizwiz Gatekeeper System Core Logic (State Machine)
 * @description 사용자의 입력 데이터를 받아 시스템적 리스크를 분석하고, 3단계 게이트키퍼 상태를 결정합니다.
 */

// --- 1. Type Definitions & Enums ---

export enum RiskLevel {
    SAFE = 'Safe',          // 초기 진단 단계 (Green Zone)
    WARNING = 'Warning',    // 임계치 초과 경고 (Yellow/Red Flash Warning)
    CRITICAL_GATEKEEPER = 'CriticalGatekeeper' // 강제 결제 유도 구간 (Hard Stop / Red Zone)
}

export enum SystemState {
    INPUT_CAPTURED,        // 사용자 입력 완료 직후
    ANALYZING,             // 시스템 분석 진행 중 (Time Pressure)
    RESULT_DISPLAYED,      // 최종 리포트 표시됨
    INTERVENTION_REQUIRED  // 게이트키퍼 액션 필요 상태
}

/**
 * @typedef {object} UserInputData - 사용자로부터 받은 기본 데이터 구조.
 * @property {string} industry - 산업 분야 (예: Finance, Healthcare)
 * @property {number} employeeCount - 직원 수
 * @property {boolean} hasComplianceCheck = true - 컴플라이언스 검사 진행 여부
 */

/**
 * @typedef {object} RiskReportPayload - 시뮬레이션 후 반환되는 최종 보고서 데이터.
 * @property {RiskLevel} riskLevel - 결정된 리스크 레벨 (Safe, Warning, Critical)
 * @property {SystemState} nextState - 시스템이 다음으로 요구하는 상태
 * @property {{score: number, reason: string}} lossMeterData - 손실 측정기 데이터 (수치화된 공포)
 * @property {boolean} isGatekeeperTriggered - 게이트키퍼 작동 여부 (결제 강제 유도)
 */

// --- 2. Mock API Simulation Function ---
/**
 * 가상의 외부 법규 DB 및 분석 엔진을 호출하여 리스크 점수를 계산하는 함수를 시뮬레이션합니다.
 * @param {UserInputData} input - 사용자 입력 데이터.
 * @returns {Promise<{score: number, reason: string}>} 계산된 리스크 점수와 근거.
 */
const mockExternalApiCall = async (input) => {
    // 3초 지연을 주어 로딩 상태를 체감하게 만듭니다. (Time Pressure 유도)
    await new Promise(resolve => setTimeout(resolve, 3000));

    let score = Math.random() * 10; // 기본 점수 부여
    let reason = "기본 시스템 검토 완료.";

    // 복잡한 비즈니스 로직을 시뮬레이션합니다. (예: 금융 산업 + 소규모 기업)
    if (input.industry === 'Finance' && input.employeeCount < 50) {
        score += 3; // 감점 요인 추가
        reason = "금융 분야의 초기 단계 기업은 내부 통제 리스크가 높습니다.";
    }

    // 컴플라이언스 체크 누락 시 가장 높은 위험도를 부여합니다. (Critical Path)
    if (!input.hasComplianceCheck || input.industry === 'Healthcare') {
        score += 8; // 매우 큰 가중치 부여
        reason = "핵심 법규(HIPAA/GDPR 등) 준수 여부가 불명확하여 치명적인 리스크가 감지되었습니다.";
    }

    // 최종 점수를 100점 만점으로 정규화하고, 임계치를 설정합니다.
    const finalScore = Math.min(Math.max(score, 20), 95);
    return { score: parseFloat(finalScore.toFixed(2)), reason };
};

// --- 3. Core State Machine Logic ---

/**
 * 사용자의 입력 데이터를 기반으로 '시스템적 생존 위협'을 분석하여 게이트키퍼 상태를 결정합니다.
 * @param {UserInputData} input - 사용자에게서 받은 초기 데이터.
 * @returns {Promise<RiskReportPayload>} 최종 리스크 보고서 페이로드.
 */
export const analyzeSystemicRisk = async (input) => {
    // 1. 분석 엔진 호출 및 대기 (State Transition: INPUT_CAPTURED -> ANALYZING)
    const lossMeterData = await mockExternalApiCall(input);

    let riskLevel;
    let isGatekeeperTriggered = false;
    let nextState = SystemState.RESULT_DISPLAYED;

    // 2. 리스크 점수에 따른 게이트키퍼 로직 실행 (The Core Logic)
    if (lossMeterData.score >= 75) {
        // [Critical Gatekeeper] - 시스템 생존에 직접 위협이 되는 경우
        riskLevel = RiskLevel.CRITICAL_GATEKEEPER;
        isGatekeeperTriggered = true;
        nextState = SystemState.INTERVENTION_REQUIRED; // 결제 강제 유도 상태로 전환
    } else if (lossMeterData.score >= 40) {
        // [Warning Gatekeeper] - 경고 단계, 즉시 액션 필요
        riskLevel = RiskLevel.WARNING;
        nextState = SystemState.INTERVENTION_REQUIRED; // 추가 진단 요청 유도 상태로 전환
    } else {
        // [Safe State] - 일단 안정적이지만, '무료 진단'을 통해 더 큰 공포를 주입할 여지 탐색
        riskLevel = RiskLevel.SAFE;
        nextState = SystemState.RESULT_DISPLAYED; // 리포트만 보여주고 끝낼 것인가? (전환율 최적화 필요)
    }

    // 3. 최종 페이로드 구성 및 반환
    return {
        riskLevel,
        nextState,
        lossMeterData: { score: lossMeterData.score, reason: lossMeterData.reason },
        isGatekeeperTriggered
    };
};

export const getRiskLevelDisplay = (level: RiskLevel): string => {
    switch(level) {
        case RiskLevel.SAFE: return "✅ Green Zone - 안전";
        case RiskLevel.WARNING: return "⚠️ Yellow/Red Flash - 경고!";
        case RiskLevel.CRITICAL_GATEKEEPER: return "🚨 Red Zone - 생존 위협 발생! 즉시 조치 필요.";
    }
};

## ✨ 컴포넌트별 애니메이션 사양 (CSS & JS Guideline)

### 1. 기본 글리치 오버레이 (`.glitch-overlay`)
*   **목적:** 페이지 전반에 걸쳐 미세한 디지털 시스템 오류의 노이즈를 부여합니다.
*   **구현 방식:** CSS Filter와 배경 애니메이션 조합.
*   **CSS Pseudo-Code:**
    ```css
    /* 1. 전역 필터 적용 (Global Scope) */
    body {
        filter: url(#chromaticAberration); /* SVG Filter 사용 권장 */
        background-image: repeating-linear-gradient(0deg, transparent, rgba(255, 0, 0, 0.03), transparent);
        animation: noise_pulse 15s linear infinite; /* 낮은 주파수의 노이즈 애니메이션 */
    }

    /* @keyframes noise_pulse { ... } : 배경에 무작위로 변하는 미세한 점 패턴을 만듭니다. */

    /* 2. 글리치 효과 구현 (Text Glitch Effect) - Critical Element에만 적용 */
    .glitch-text {
        position: relative;
        font-family: 'Roboto Mono', monospace;
        animation: glitch-anim 0.1s infinite alternate step-end; /* 매우 짧고 불규칙하게 반복 */
    }

    /* ::before 와 ::after Pseudo-elements를 이용해 색상 왜곡(Chromatic Aberration)을 만듭니다. */
    .glitch-text::before, .glitch-text::after {
        content: attr(data-text); /* 원본 텍스트 반복 사용 */
        position: absolute;
        top: 0; left: 0; width: 100%; height: 100%;
        /* 색상 분리 및 오프셋 효과를 통해 아날로그적 노이즈 느낌 부여 */
    }
    ```

### 2. 경고 발생 (Critical Alert/Red Flash)
*   **트리거:** 리스크 지표가 임계치에 도달하거나, 핵심 카피(예: 'CRITICAL LEGAL EXPOSURE DETECTED')를 만났을 때.
*   **애니메이션 흐름 (JavaScript Control):**
    1.  **(Pre-Flash):** 경고 요소(`div`)의 투명도를 점진적으로 높이며 글리치 효과가 증가합니다. (`opacity: 0` $\rightarrow$ `opacity: 0.8`).
    2.  **(Impact Flash):** 짧은 순간(50ms) 동안 배경 전체에 강렬한 `#C0392B` 플래시를 적용합니다. (CSS Class Toggle: `.red-flash`).
    3.  **(Warning State):** 플래시 직후, 경고 메시지 텍스트가 **글리치 효과와 함께 깜빡이면서(Flicker)** 화면 중앙에 강하게 고정됩니다. 이때 `setInterval`을 이용해 무작위로 색상 왜곡(`text-shadow`)을 토글하여 불안감을 유지합니다.
*   **CSS Pseudo-Code:**
    ```css
    /* 1. 플래시 효과 (매우 짧게, 순간적인 충격) */
    @keyframes red-flash {
        0% { background-color: transparent; opacity: 1; }
        50% { background-color: rgba(192, 57, 43, 0.8); opacity: 1; } /* Red Zone 컬러 사용 */
        100% { background-color: transparent; opacity: 1; }
    }
    /* JS가 .red-flash 클래스를 추가/제거하며 타이밍을 제어해야 합니다. */

    /* 2. 깜빡이는 경고 메시지 (Flickering Alert) */
    @keyframes flicker {
        0%, 19%, 21%, 23%, 25%, 54%, 56%, 100% { opacity: 1; }
        20%, 24%, 55% { opacity: 0.8; text-shadow: none; } /* 간헐적인 깜빡임 */
    }
    ```

### 3. 손실액($Y$) 수치화 및 데이터 로딩 (`QLoss Display`)
*   **목적:** 숫자를 단순 보고가 아닌, 시스템이 '계산하는 위협'으로 보이게 합니다.
*   **애니메이션 흐름 (JS Control):**
    1.  **(Pre-Load):** "ANALYZING SYSTEM INTEGRITY..." 와 같은 코드가 빠르게 스크롤되는 텍스트를 배경에 배치합니다 (Code Ticker Effect).
    2.  **(Data Load):** 손실액($Y$) 수치(예: $3,450,000)는 일반적인 `+=` 로직이 아닌, **'불규칙한 카운트업 애니메이션'**을 사용합니다. 마치 시스템이 불안정하게 계산 값을 산출해내는 듯한 느낌을 줘야 합니다 (Randomized Increment).
    3.  **(Display):** 최종 숫자가 나타날 때 글리치 효과를 한 번 더 짧게 적용하여 '확정된 위협 수치'임을 각인시킵니다.

*   **발생 메커니즘 (Process Gap):** 내부 매뉴얼이나 규정(법규)이 업데이트되었음에도 불구하고, 실제 업무 흐름(워크플로우)에 **'자동 재검토 및 강제 삽입' 로직**이 없어 담당자의 기억과 판단에 의존하게 됨. 특정 단계의 필수 승인 절차 누락 발생.
*   **규정 위반 내용:** 산업 표준 규정, 전문 서비스 윤리 규정(Professional Ethics) 등 법적 체크포인트 미준수. (구체적인 벌금액 산정보다 **'사업 지속 불가능 리스크'**가 핵심).
*   **정량화된 손실/벌금 추정치:**
    *   **직접 비용:** 프로젝트 지연으로 인한 계약 위반 페널티, 평균 수천만 원 ~ 수억 원.
    *   **잠재적 손해액($Y$):** 규제 당국의 감사를 통과하지 못하여 사업 운영 자체가 일시 정지되는 **'운영 중단 비용(Business Interruption Loss)'**, 최대 연간 수익의 10% 이상.

#### Case 3: AI 환각으로 인한 '준전문가 책임' 리스크
*   **위험 유형:** LLM 기반 지식 검색 및 추론 과정에서의 근거 제시 실패 (Hallucination).
*   **대상 전문직/산업:** 법률, 컨설팅, 금융 심사.
*   **발생 메커니즘 (Process Gap):** AI 시스템이 내부의 분산된 지식(Knowledge Silo)을 통합적으로 참조하지 못하고, 그저 가장 그럴듯한 **'문장적 유사성'**에 기반하여 허위 정보를 생성함. 답변과 함께 반드시 법적/규정적 근거 링크 및 출처를 강제하는 시스템이 없음.
*   **규정 위반 내용:** '준전문가 책임(Quasi-professional Liability)' 발생. (법률 자문 등 전문 영역에서 잘못된 조언을 제공하여 의뢰인에게 손해).
*   **정량화된 손실/벌금 추정치:**
    *   **평균 배상액:** 단순 컨설팅 오류가 아닌, 법적 책임이 걸린 경우 최소 수천만 원 이상의 직접적인 민사소송 비용 발생.
    *   **신뢰도 하락(무형 자산):** 전문 서비스의 가장 큰 무기인 '신뢰' 자체가 붕괴되어 시장에서 퇴출될 위험 (가장 높은 리스크).

## II. 핵심 리스크 사례 분석 (Quantified Risk Report)

### 📄 Case 1: PII 마스킹 실패에 따른 데이터 유출 사고
*   **발생 원인:** [Process Gap] 데이터 전처리/AI 학습 과정에서 **'강제 비식별화 게이트웨이(Mandatory De-identification Gateway)'** 부재. 수동 검토 및 체크리스트 의존.

*   **손실 시뮬레이션 ($Y$):** 평균 벌금 $A \text{ Million} + (\text{매출액} \times 3\%)$의 조합적 손해.

### 📄 Case 2: 컴플라이언스 게이트웨이 부재에 따른 프로세스 드리프트
*   **발생 원인:** [Process Gap] 법규/내부 매뉴얼 업데이트와 실제 업무 흐름(Workflow) 간의 **자동 연동 및 강제 검증 시스템** 미구축. 사람이 '기억'하는 단계에서 필수 승인 절차 누락.
*   **위반 규정:** 산업 표준, 전문 서비스 윤리 규정 위반 (운영 중단 리스크).
*   **손실 시뮬레이션 ($Y$):** 직접 페널티 $B \text{ Million} + (\text{연간 수익} \times 10\%)$의 운영 중단 비용.

### 📄 Case 3: AI 환각(Hallucination)으로 인한 준전문가 책임
*   **발생 원인:** [Process Gap] LLM 기반 답변에 대한 **'출처 강제 제시 및 출처 검증 게이트웨이'** 미구축. 단순 패턴 인식 오류를 법적 근거로 오해하여 결과물 생성.
*   **위반 규정:** 준전문가 책임(Quasi-professional Liability) 발생 위험. (법률/의료 자문의 경우).

## III. 컴포넌트별 상세 스펙 (Component Breakdown)

### 1. 전역 오버레이 및 배경 (Global Overlay & Background)
*   **배경:** `#1A1A1A`
*   **필수 적용 필터:** `Noise/Glitch Effect` (CSS Filter).
    *   전체 페이지에 투명도(Opacity) 5%의 **미세한 색상 채널 분리(Chromatic Aberration)**와 저주파 노이즈 패턴을 지속적으로 오버레이합니다. (`background-image: repeating-linear-gradient(...)`).
    *   이는 사용자가 '깨끗한 화면'이라는 안정감을 느끼지 못하게 하여, 시스템의 불안정성을 전제합니다.

### 2. 리스크 지표 섹션 (The Core Data Panel) - 가장 중요
이 영역은 마치 서버 대시보드의 모니터링 패널처럼 설계되어야 합니다.

*   **레이아웃:** 그리드(Grid) 기반, 각 지표를 독립된 `Card` 컴포넌트로 분리합니다.
*   **지표 형태:** **[Metric Label] : [Data Value] / [Threshold Value]** (예: *PII Leakage Risk*: 72% / 60%).
*   **애니메이션 스펙 (JS Trigger):**
    1.  **데이터 로드 시:** 모든 수치(`[Data Value]`)는 일반적인 페이드인 대신, **'숫자 카운트업(Number Counter Animation)'** 효과를 사용하여 마치 시스템이 실시간으로 데이터를 측정하는 것처럼 보이게 합니다. (예: 0 $\rightarrow$ X%로 빠르게 상승).
    2.  **임계치 초과 감지 (The Trigger):** 자바스크립트가 `[Data Value]`가 `[Threshold Value]`를 초과했음을 감지하면, 해당 `Card` 전체와 주변 섹션에 다음을 트리거합니다.
        *   **A. Red Flash:** 배경색이 200ms 동안 강렬한 `#C0392B` 플래시로 순간적으로 변함 (Opacity 0 $\rightarrow$ 1 $\rightarrow$ 0).
        *   **B. 경고 문구 깜빡임:** `Card` 제목 또는 지표 주변에 "!!! CRITICAL ALERT !!!"라는 텍스트가 무작위 간격으로 짧게, 강하게 깜빡이는 애니메이션(`text-shadow: 1px 0 red, -1px 0 blue;`)을 적용합니다.

### 3. 경고 시스템 모달 (The Pop-up Modal)
임계치 초과가 발생했을 때 화면 전체를 가로막는 '시스템 강제 알림' 창입니다.

*   **출현 조건:** 모든 핵심 리스크 지표 중 **어느 하나라도 임계치를 초과할 경우**, 무조건 팝업되어야 합니다.
*   **디자인:** 중앙에 위치한 사각형 모달(Modal). 경고등처럼 디자인하며, 상단에 `[SYSTEM ALERT]` 배지(Badge)를 배치합니다.
*   **내용 구성:**
    1.  **헤드라인 (H1):** **`⚠️ CRITICAL SYSTEM FAILURE DETECTED: [리스크 유형]`** (폰트: Roboto Mono, 색상: `#C0392B`, 글리치 효과 필수).
    2.  **본문 (P):** "현재 시스템의 운영 상태는 법적 임계치를 초과했습니다. 즉각적인 수동 개입(Manual Intervention)이 필요합니다." (권위적인 문체 유지).
    3.  **CTA:** `[Diagnosis Request]` 버튼을 배치하되, 이 버튼은 일반적인 버튼 모양이 아닌, **'Reboot/Override'**를 연상시키는 직사각형의 시스템 제어 패널처럼 보이게 디자인합니다.

## IV. 기술 구현 브리프 (Developer Implementation Brief)
*   **CSS:** 모든 애니메이션(Glitch, Flash)은 CSS `keyframes`와 `:hover`/`:focus` 상태 변화로 구현하는 것을 원칙으로 합니다. JavaScript는 오직 **'임계치 감지 및 플래시 트리거'** 로직에만 사용합니다.
*   **JS (핵심):**
    1.  데이터 API를 통해 리스크 점수(Score)를 받는다.
    2.  `if (score > threshold)` 조건 충족 시, `document.getElementById('main-container').classList.add('critical-fail');` 명령을 실행하여 전역 애니메이션 클래스를 추가한다.
*   **파일 구조 제안:** 이 템플릿은 React 또는 Vue와 같은 컴포넌트 기반 프레임워크를 사용하여 재사용성을 극대화해야 합니다.

## 2. 핵심 컴포넌트 A: Red Zone Glitch Noise System (글리치 노이즈)

### A-1. Glitch Overlay (전역 노이즈)
*   **효과:** 화면의 특정 영역(Red Zone 섹션 진입 시점)에 미세하고 불규칙적인 색상 왜곡 및 노이즈가 발생합니다.
*   **CSS Spec:**
    ```css
    /* 1. Base Glitch Filter */
    .glitch-overlay::before,
    .glitch-overlay::after {
        content: '';
        position: absolute;
        top: 0; left: 0; width: 100%; height: 100%;
        background: rgba(256, 30, 43, 0.1); /* Red Zone 투명도 */
        pointer-events: none;
        animation: glitch-move 0.5s infinite linear alternate;
    }

    /* 2. Chromatic Aberration (색상 분리) */
    .glitch-overlay::before {
        transform: translate(-1px, -1px);
        mix-blend-mode: screen; /* 색상 간섭 효과 극대화 */
        animation-delay: 0s;
    }

    .glitch-overlay::after {
        transform: translate(2px, 2px);
        mix-blend-mode: difference; /* 대비 효과 증폭 */
        animation-delay: -0.1s;

    /* 3. Keyframes (불규칙적 움직임) */
    @keyframes glitch-move {
        0% { transform: translate(0, 0); opacity: 0.2; }
        50% { transform: translate(-3px, 3px); opacity: 0.4; }
        100% { transform: translate(3px, -3px); opacity: 0.2; }
    }
    ```

## 3. 핵심 컴포넌트 B: Loss Meter (QLoss) 애니메이션 스펙

**적용 범위:** 랜딩 페이지 중앙의 데이터 시각화 섹션. 가장 공포감을 극대화하는 곳입니다.
**구현 방식:** SVG `<path>` 요소와 JavaScript 기반의 상태 변화에 따른 애니메이션 제어.

### B-1. Loss Meter 구조 (SVG)
*   **요소:** 하나의 가로 막대 형태의 경로(Path)를 사용합니다. `d` 속성을 통해 현재 손실액을 시각화하며, 이 경로는 '안전' 영역에서 '위험' 영역으로 이동하는 애니메이션이 필수입니다.
*   **애니메이션 로직 (JS Trigger):**
    1.  **Initial State:** Path가 낮은 값(Safe Loss)부터 시작합니다. (`d="M 0,20 L 50,20"`)
    2.  **Risk Escalation:** 사용자가 리스크를 인지하는 순간, 스크립트가 `setInterval`을 통해 경로의 끝점(`L X, Y`)을 점점 더 오른쪽(Loss Accumulation)으로 이동시킵니다. (속도: 가속도가 붙는 느낌).
    3.  **Critical Threshold Hit:** 손실액이 임계치($Y_{critical}$)에 도달하는 순간, Path가 빨간색(`stroke:#C0392B`)으로 변경되고, **'삐-' 소리와 함께 짧은 진동 애니메이션(Shake)**을 동반해야 합니다.

### B-2. Loss Meter Color Mapping (Criticality)
| 리스크 상태 | 범위 (Loss) | 색상 코드 | 시각적 효과 | [근거: Self-RAG] |
| :--- | :--- | :--- | :--- | :--- |
| **SAFE** (안전) | 0 ~ $X_{low}$ | `#2ECC71` (Green) | 부드러운 선형 애니메이션. | [추측] |
| **WARNING** (주의) | $X_{low}$ ~ $Y_{warning}$ | `yellow` (Amber/Warning) | 주기적인 미세 진동(Pulse effect). | [근거: Self-RAG] |
| **CRITICAL** (위험) | $> Y_{warning}$ | `#C0392B` (Red Zone) | 강한 깜빡임, 시스템 경고 사운드 동반, Path가 떨리는 효과(Shake). | [근거: Self-RAG] |

## 4. 핵심 컴포넌트 C: Gatekeeper Flow Logic & UI

**목표:** 리스크 진단 결과 (Loss Meter) $\rightarrow$ 임계치 도달/경고 발생 $\rightarrow$ **강제 구매 요청 모달**로의 흐름을 구축합니다.
**원칙:** 결제 망설임 접점마다 공포감과 긴급성을 극대화하는 마이크로카피가 삽입되어야 합니다.

### C-1. 게이트키퍼 트리거 (JS Logic)
*   **Trigger:** Loss Meter 값이 `Y_{warning}`을 초과하고, 사용자가 '진단 보고서 다운로드' 버튼을 클릭했을 때 발동합니다.
*   **State Transition:**
    `[진단 결과 표시] $\rightarrow$ [🚨 Red Zone 경고 발생 (글리치 활성화)] $\rightarrow$ [구매 필수 모달 강제 오픈]`

### C-2. 게이트키퍼 모달 스펙
*   **스타일:** 일반적인 팝업이 아닌, **'시스템 오류/긴급 공지(System Alert)'** 창처럼 보이게 디자인합니다. (배경에 글리치 오버레이 필수).
